## 1. システム概要

### 何をするシステムか

特定のウェブサイトへの自動ログイン・CSVダウンロード・Googleドライブへの格納・スプレッドシートへの統合を、複数サイトに対して定期実行する仕組み。

- 対象サイトにログインし、CSVをダウンロード
- ダウンロードしたCSVをサイトごとに指定のGoogleドライブフォルダへ格納
- 複数のCSVを1つのスプレッドシートにタブ別で統合
- インポート完了後、Driveの元CSVファイルを自動削除
- 稼働状況（最終稼働日時）をスプレッドシートに自動記録し、エラー把握を容易にする

### なぜこの構成か

| 課題 | 解決策 |
|------|--------|
| 属人化の防止 | スプレッドシートで全設定を一元管理。担当者変更時も運用継続可能 |
| 複数サイト対応 | 行を追加するだけで対象サイトを増やせる |
| CSV格納先の柔軟な切り替え | H列にサイトごとのDriveフォルダURLを指定 |
| データの一元集約 | I列のスプレッドシートにタブ別で自動統合 |
| エラー検知のしやすさ | B列（最終稼働日時）で最後に正常稼働した日時を確認可能 |
| リポジトリのパブリック公開 | 認証情報をコードに含めない設計。スプレッドシートURLはGitHub Secretsに格納 |

## 2. 設計方針

- **認証情報（ログインID・パスワード）はスプレッドシート内に保持する**  
  スプレッドシートは社内メンバーのみ閲覧可能な前提。組織ドメインで共有範囲を制限すること。

- **スプレッドシートのURLはGitHub Secretsに格納する**  
  リポジトリをパブリックにしても、URLがソースコードやログに露出しない。

- **スクリプトのロジック（操作内容）はGitHubリポジトリで管理する**  
  認証情報を含まない操作ロジックのみをコードとして管理。パブリックリポジトリでも安全。

- **ON/OFFの制御はA列で完結させる**  
  A列をOFFにするだけで、該当サイトへの自動化を停止。コードを触る必要はない。

- **B列（最終稼働日時）はシステムが自動で書き込む**  
  ワークフロー正常完了後にGASへコールバックし、B列に日時を記録。人間が手動で書く列ではない。

## 4. コンポーネント詳細

### 4-1. Googleスプレッドシート
- **役割:** システム全体の設定台帳。コードを触らずに運用変更できる唯一の操作ポイント。
- 閲覧・編集権限は組織ドメイン内のメンバーのみに限定
- URLはGitHub Secretsに格納し、外部から特定できない状態を維持
- B列（最終稼働日時）とH列（ドライブURL）、I列（リード格納URL）はGASが書き込む

### 4-2. GAS（Google Apps Script）
- **役割:** スプレッドシートの定時読み取りとGitHub APIの呼び出し、および完了後のステータス書き込み。
- GASのトリガー機能（時間ベース）で毎日特定時刻に自動実行
- A列がONの行のみを対象にループ処理
- 各行のデータをinputsとしてGitHub APIに渡す
- GitHubAPIを叩くためのPAT（Personal Access Token）はGASのスクリプトプロパティ（`PropertiesService`）に格納する

### 4-3. GitHub Actions ワークフロー
- **役割:** 自動化の実行環境。
- トリガー: `workflow_dispatch`（GASからのAPI呼び出しで起動）
- inputsで受け取ったデータを環境変数としてPlaywrightスクリプトに渡す
- `SPREADSHEET_URL`などの機密値は`${{ secrets.XXXX }}`形式で参照し、ログに出力しない
- Ubuntu（ubuntu-latest）上でNode.js + Playwrightを実行
- 正常完了後にGASのコールバックエンドポイントへ通知

### 4-4. スクリプト群（scripts/ディレクトリ）
- **役割:** サイトごとの操作ロジック。
- Playwrightを使ったブラウザ自動化スクリプト
- 1サイト = 1スクリプトファイルを基本とする
- 認証情報は環境変数経由で受け取る（スクリプト内にハードコードしない）

## 5. スプレッドシート仕様

**シート名:** `automation_config`

| 列 | 列名 | 書き込み主体 | 説明 | 例 |
|----|------|------------|------|-----|
| A | 稼働ステータス | 人間（手動） | ONにすると自動化対象になる | ON |
| B | 最終稼働日時 | GAS（自動） | ワークフロー正常完了時に書き込まれる | 2026-06-30 08:12 |
| C | サイト名 | 人間（手動） | 管理用の識別名（自由記述） | 社内ポータル |
| D | 対象URL | 人間（手動） | ログインページや操作対象のURL | https://example.com/login |
| E | ログインID | 人間（手動） | 対象サイトのログインID | user@example.com |
| F | パスワード | 人間（手動） | 対象サイトのパスワード | p@ssw0rd |
| G | スクリプトパス | 人間（手動） | GitHubリポジトリ内の実行スクリプトの相対パス | scripts/siteA.js |
| H | ドライブURL | 人間（手動） | CSVの格納先GoogleドライブフォルダのURL | https://drive.google.com/drive/folders/xxxxx |
| I | リード格納URL | 人間（手動） | CSV統合先スプレッドシートのURL | https://docs.google.com/spreadsheets/d/xxxxx |

### 運用ルール
- 1行目はヘッダー行とし、GASは2行目以降を処理対象とする
- A列をOFFにした行はGASが完全にスキップする
- B列は手動で書き換えない（システムが管理する列）

## 6. CSVダウンロード・格納・統合フロー

```
[GitHub Actions上でPlaywright実行]
        │
        ▼
対象サイトにログイン → CSVをダウンロード（ランナー上の一時ディレクトリに保存）
        │
        ▼
GoogleドライブへアップロードAPIで送信
  → H列で指定されたフォルダに格納
        │
        ▼
Google Sheets APIでI列のスプレッドシートにインポート
  → サイト名（C列）をタブ名として使用
  → 既存タブがあれば上書き、なければ新規作成
        │
        ▼
GoogleドライブAPIでアップロードしたCSVファイルを削除
        │
        ▼
GASコールバック → B列に最終稼働日時を書き込み
```

### タブ名のルール
- タブ名 = C列のサイト名をそのまま使用
- タブ名に使えない文字（`/ \ ? * [ ] :` など）は自動的に `_` に置換

## 7. GitHub Secrets 管理

| Secret名 | 内容 |
|----------|------|
| `SPREADSHEET_URL` | 管理台帳スプレッドシートのURL |
| `GAS_PAT` | GitHub PAT（GASがワークフローを起動するためのトークン） |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | GoogleドライブとSheets APIを操作するためのサービスアカウントJSON |

### PATのスコープ設定
- Fine-grained PATを使用し、対象リポジトリを限定
- 必要なPermission: **Actions: Write のみ**

### Google Service Account
- Google Cloud Consoleでサービスアカウントを作成
- Drive API と Sheets API を有効化
- H列のDriveフォルダとI列のスプレッドシートに対して編集権限を付与

## 9. 処理フロー詳細

```
[毎日 定時（例: 08:00 JST）]
        │
        ▼
GASトリガー起動 → スプレッドシートを読み取り → A列=ONの行を順番に処理
        │
        ├── 行データをinputsに格納
        │     ├── site_name        : C列
        │     ├── target_url       : D列
        │     ├── login_id         : E列
        │     ├── password         : F列
        │     ├── script_path      : G列
        │     ├── drive_folder_url : H列
        │     └── lead_sheet_url   : I列
        │
        ▼
GitHub API（workflow_dispatch）を呼び出し
POST /repos/{owner}/{repo}/actions/workflows/auto.yml/dispatches
        │
        ▼
GitHub Actions ワークフロー起動
        ├── Node.js + Playwright セットアップ
        ├── G列（script_path）のスクリプトを実行
        │
        ▼
Playwrightスクリプト（例: scripts/siteA.js）
        ├── 対象URLにアクセス → ログイン → CSVダウンロード
        │
        ▼
lib/drive-uploader.js → H列のDriveフォルダにCSVをアップロード
        ▼
lib/sheets-importer.js → I列のスプレッドシートにタブ別インポート → 元CSVを削除
        ▼
lib/gas-callback.js → GASへ完了通知
        ▼
GAS（コールバック受信） → B列に最終稼働日時を書き込み（YYYY-MM-DD HH:mm）
```

## 10. セキュリティ設計

| 情報 | 保存場所 | 外部から見えるか |
|------|----------|----------------|
| ログインID・パスワード | スプレッドシート（社内限定） | ❌ 見えない |
| スプレッドシートURL | GitHub Secrets | ❌ 見えない |
| GitHub PAT | GitHub Secrets + GASスクリプトプロパティ | ❌ 見えない |
| GoogleサービスアカウントJSON | GitHub Secrets | ❌ 見えない |
| DriveフォルダURL（H列） | スプレッドシート（社内限定） | ❌ 見えない |
| リード格納URL（I列） | スプレッドシート（社内限定） | ❌ 見えない |
| 操作ロジック（スクリプト） | GitHubリポジトリ（パブリック） | ✅ 見える（認証情報は含まない） |
| ワークフロー定義（yaml） | GitHubリポジトリ（パブリック） | ✅ 見える（Secrets参照のみ） |

## 11. 運用ルール

### 日常操作

| やりたいこと | 操作 |
|------------|------|
| 特定サイトの自動化を一時停止 | A列をOFFに変更 |
| 新しいサイトを追加 | スプレッドシートに1行追加 ＋ scripts/に操作スクリプトを追加 |
| CSV格納先フォルダを変更 | H列のURLを変更 |
| データ統合先スプレッドシートを変更 | I列のURLを変更 |
| スクリプトの操作内容を変更 | GitHubの該当スクリプトを編集 |
| エラーが出ていないか確認 | B列の最終稼働日時が古い行をチェック |
| 実行時刻を変更 | GASのトリガー設定を変更 |

### 禁止事項
- スクリプト内への認証情報のハードコード
- スプレッドシートの共有範囲を「リンクを知っている全員」に変更すること
- B列を手動で書き換えること（システムが管理する列）
- `workflow_dispatch`のトリガー削除
